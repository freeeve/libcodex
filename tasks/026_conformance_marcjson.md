# 026 — Conformance: MARC-in-JSON

Verify the JSON layout matches the de-facto MARC-in-JSON structure other tools
read and write (there is no ISO standard; the reference is the community spec).

## References
- MARC-in-JSON proposal (Ross Singer):
  https://web.archive.org/web/*/http://dilettantes.code4lib.org/blog/2010/09/a-proposal-to-serialize-marc-in-json/
- pymarc's JSON output and `marc-record-js` / Bill Dueber's marc-hash as
  interoperability references.

## Checks
- Top level: `{"leader": "...", "fields": [ ... ]}` with fields in record order.
- Control field: `{"001": "value"}` (a single-key object, string value).
- Data field: `{"245": {"ind1": "1", "ind2": "0", "subfields": [{"a": "..."}]}}`.
- `subfields` is an ordered array of single-key objects; repeated codes preserved.
- Indicators serialized as single-character strings (blank as a space).
- Decoder tolerates key ordering and ignores unknown keys.

## Verification
- Round-trip against pymarc-produced JSON for the same records (structural equality
  after decode, and pymarc can read our output).
- Add fixtures generated by pymarc to `marcjson/testdata`.

## Acceptance
- Documented structural conformance with interoperability fixtures from pymarc.
